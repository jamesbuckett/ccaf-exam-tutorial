Setup · venv

Enabling venv to run the tutorials

  • Every tutorial installs its packages into a throwaway .venv — each tutorial's SDK version stays isolated.
  • Keeps system Python clean and sidesteps PEP 668 errors on Ubuntu 23.04+ and current Homebrew.
  • Do the whole dance once on Tutorial 01, then repeat the short version for every later tutorial.
  • Activation is per-shell — the (.venv) prompt prefix is the signal it's live.
  • Windows PowerShell activates with .\.venv\Scripts\Activate.ps1 instead.
  • Hit an error? Every venv symptom has a dedicated troubleshooting entry with a fix and a reason.

PEP 668 is the rule that makes system Python refuse global pip install to protect OS packages. If PowerShell blocks activation on "execution policy", run Set-ExecutionPolicy -Scope Process -ExecutionPolicy Bypass in that shell first. The dedicated entries: ensurepip is not available, externally-managed-environment, ModuleNotFoundError on an already-installed package.

Setup · venv

Verify, deactivate, re-enter (steps 4–5)

which python3 pip              # both must point inside .venv/

Tutorial 01 · Domain 1

The Agentic Loop — think / act / observe with the Anthropic SDK

Concept covered

  • An agent is a Claude instance wrapped in a loop that calls tools, observes results, and decides the next action until a stop condition fires.
  • stop_reason is the gate — tool_use goes back around, end_turn exits. MAX_ITERATIONS is the last-resort seatbelt.

You'll build

  • A think → act → observe loop from scratch
  • stop_reason flipping from tool_use to end_turn
  • A runaway loop, by removing the termination cap
  • Each turn's tool_use / tool_result round-trip

Source: Domain 1, the Agentic Loop mental model in the companion study guide. Each turn is one client.messages.create() call. Prerequisites: Python 3.11+, anthropic SDK 0.40+, an exported API key — nothing else, it runs locally. Before starting, record a spend baseline at console.anthropic.com/usage so you can check the delta afterwards.

Tutorial 01 · loop.py

The catalogue and the cap (1/3)

client = anthropic.Anthropic()   # reads ANTHROPIC_API_KEY
MODEL = "claude-sonnet-4-6"
MAX_ITERATIONS = 5               # the termination cap

tools = [{
    "name": "get_weather",
    "description":
        "Return the current temperature in Celsius for a given city.",
    "input_schema": {"type": "object",
                     "properties": {"city": {"type": "string"}},
                     "required": ["city"]},
}]
messages = [{"role": "user",
             "content": "Compare the weather in Paris and Tokyo right now."}]

Trimmed here: import anthropic, json, os and the run_tool dispatcher — a pure-Python function returning fake temperatures for Paris, Tokyo, and Cape Town. The model never executes run_tool; you do. MAX_ITERATIONS is the single most important line in the file: without it the loop is while True and the bill grows without bound. Three tool fields matter: name, description (the routing signal), input_schema (guaranteed by the API only with strict: true).

Tutorial 01 · loop.py

Think, then the exit gate (2/3)

for turn in range(MAX_ITERATIONS):
    resp = client.messages.create(
        model=MODEL, max_tokens=1024, tools=tools, messages=messages
    )
    print(f"--- turn {turn} · stop_reason={resp.stop_reason} ---")
    messages.append({"role": "assistant", "content": resp.content})

    if resp.stop_reason == "end_turn":
        print("FINAL:", resp.content[-1].text)
        break

python · "think" is the create() call; end_turn is the healthy exit

resp.content[-1].text grabs the prose answer — the last block, because earlier blocks might be thinking text. The break means the for/else clause on the next slide is not executed; else on a for only fires when the loop exhausts without break.

Tutorial 01 · loop.py

Act, observe, and the safety cap (3/3)

    tool_results = []
    for block in resp.content:
        if block.type == "tool_use":
            result = run_tool(block.name, block.input)
            tool_results.append({
                "type": "tool_result",
                "tool_use_id": block.id,
                "content": json.dumps(result),
            })
    messages.append({"role": "user", "content": tool_results})
else:
    print("!! loop exhausted without end_turn — safety cap hit")

python · "act" is run_tool; "observe" is appending the tool_result

Iteration-cap exit — the else branch — means something went wrong; the model wanted another turn and you cut it off. Tutorial 01's build-and-break triggers exactly this on purpose.

Tutorial 01 · Code trace

What happens when you run it

  • Turn 0 · think — the model can't answer without the tool: stop_reason="tool_use" plus a Paris ToolUseBlock.
  • Act and observe share the same Python iteration: run_tool executes, the result rides back as a tool_result.
  • tool_use_id must echo the block's id; content must be a string — the first BadRequestError readers hit.
  • Turn 1 — Tokyo. Sonnet may batch both calls in turn 0; the for block loop handles that identically.
  • Turn 2 · end — end_turn, prose answer, break; the for/else fires only when the cap exhausts.
  • The model has no memory between calls — every create() re-sends the full messages list.

Before running, walk it in your head: one get_weather tool, two cities, so the model must call twice. HTTP call #1 returns the Paris request; call #2 carries the Paris result and returns the Tokyo request; call #3 has both temperatures and composes the answer. The loop's job is really just to accrete history.

Tutorial 01 · Code trace

The final messages ledger — six entries

[0] user       "Compare the weather in Paris and Tokyo right now."
[1] assistant  [ToolUseBlock(name='get_weather', input={'city': 'Paris'})]
[2] user       [tool_result(Paris → {'temp_c': 14})]
[3] assistant  [ToolUseBlock(name='get_weather', input={'city': 'Tokyo'})]
[4] user       [tool_result(Tokyo → {'temp_c': 22})]
[5] assistant  [TextBlock("Paris is 14 °C while Tokyo is 22 °C…")]

Tool results ride inside user messages — there is no third role. State lives in your Python process, not on Anthropic's side.

Two things to internalise from this shape: the model has no memory between calls (every create() re-sends the whole list), and from the model's perspective "you told me something" and "the tool told me something" are both inputs from the user role. The loop is a three-state machine — think (create), act (run_tool), observe (append tool_result) — that terminates on end_turn, an iteration cap, or an error.

Tutorial 01 · Walkthrough

Three runs, three observations

  • Run python loop.py — two turns with stop_reason=tool_use, then end_turn and prose.
    • The think → act → observe cycle is explicit; the API never hides it.
  • Add print(resp.content[0]) after the turn header — a ToolUseBlock on tool turns, a TextBlock at the end.
    • Tool use is a structured response block, not parsed text — the contract you code against.
  • Print the role list after the loop — user → assistant → user → … alternating.
    • The loop builds the history; the model keeps nothing between calls.

Step 3's exact line: print(json.dumps([{'role': m['role']} for m in messages], indent=2)) appended after the for/else block. Expected: user, assistant, user, assistant, user, assistant with tool_use / tool_result alternating inside.

Tutorial 01 · Build-and-break

Starve the loop of iterations

  • Edit 1: MAX_ITERATIONS = 1 — was 5, deliberately too low.
  • Edit 2: widen the prompt to three cities — Paris, Tokyo, and Cape Town — so one turn can't finish.
  • Expected failure: exit via the else branch — !! loop exhausted without end_turn.
  • Confirm: the final printed stop_reason is tool_use, not end_turn.
  • Revert: restore MAX_ITERATIONS = 5.

The model wanted another turn; you cut it off. This is the same mechanism that protects you when a loop genuinely runs away — watch it fire once on purpose and you'll never ship an unbounded loop.

Tutorial 02 · Domain 1

Hub-and-Spoke Orchestration with Isolated Sub-agent Context

Concept covered

  • One orchestrator delegates independent slices of work to sub-agents running in their own context windows.
  • Sub-agents return summaries, not transcripts — and there are no worker-to-worker edges. That's how the hub's context stays bounded.

You'll build

  • spawn_subagent(task_brief) → summary
  • Proof the parent holds no sub-agent transcripts
  • The leaky-context bug, triggered on purpose
  • Token counts on parent vs child — the isolation boundary

Source: Domain 1, Hub-and-Spoke model, Scenario 03 (Multi-Agent Research). The orchestrator sees everything; each sub-agent sees only its slice. Direct edges between workers reintroduce the context-bloat problem hub-and-spoke was built to solve. Prereqs: Tutorial 01 complete, same Python environment and key. Record a spend baseline first.

Tutorial 02 · orchestrator.py

The spawn — one isolated conversation per child

def spawn_subagent(task_brief: str, subagent_name: str) -> str:
    """Run an isolated sub-agent; return a summary string only."""
    resp = client.messages.create(
        model=MODEL, max_tokens=512,
        system=(f"You are a {subagent_name} sub-agent. Do the task "
                "briefly and return ONE paragraph of findings. "
                "Do not narrate your process."),
        messages=[{"role": "user", "content": task_brief}],
    )
    return resp.content[0].text

python · the system line is the summary contract; messages holds only the brief

Trimmed: the [spawn] and [done] print lines that log each child's character and token counts. Sub-agent identity lives in the system prompt, not in history — which is why each child starts fresh. Reuse Tutorial 01's .venv: cd agentic-loop and source .venv/bin/activate.

Tutorial 02 · orchestrator.py

The hub — three spawns, one synthesis

def orchestrate(user_goal: str) -> str:
    plan = [
        ("researcher",  f"Find 3 key facts relevant to: {user_goal}"),
        ("critic",      f"List 2 likely counter-arguments to: {user_goal}"),
        ("synthesizer", f"Suggest a 1-sentence thesis for: {user_goal}"),
    ]
    findings = {name: spawn_subagent(brief, name) for name, brief in plan}

    resp = client.messages.create(
        model=MODEL, max_tokens=512,
        system="You are an orchestrator. Combine sub-agent findings into a brief.",
        messages=[{"role": "user", "content": f"Goal: {user_goal}\n\n"
                   + "\n\n".join(f"[{k}]\n{v}" for k, v in findings.items())}],
    )
    return resp.content[0].text

Run with: print(orchestrate("Should a startup adopt Kubernetes on day one?")). The plan list is a static three-tuple of (role, task brief); the spawns run sequentially — swap in the async client and asyncio.gather to parallelise. The final create() concatenates summaries, never transcripts.

Tutorial 02 · Code trace

Four independent create() calls

  • Three spawns, then one synthesis at the hub — none of the four share a conversation.
  • Each child gets its own empty messages and its own system prompt; the researcher never sees the user's original goal, only its pre-framed brief.
  • Isolation buys genuinely independent critique — the critic's counter-arguments aren't biased by the researcher's facts.
  • The hub reads summaries only: ~500–700 input tokens. Full transcripts would be 3–10× larger.
  • At N=10 sub-agents, that difference is the whole tutorial.

Hub-and-spoke = N isolated conversations plus one synthesis conversation. Workers never talk to each other; the hub only ever receives summaries. Every deviation from that shape — leaking transcripts, adding worker-to-worker edges — is a Domain 1 anti-pattern.

Tutorial 02 · Walkthrough

Watch the token boundary

  • Run python orchestrator.py — three [spawn]/[done] pairs, then an [orchestrator] token report.
  • Extend the [done] log with input_tokens: each sub-agent runs 50–150 in; the orchestrator ~600 — three summaries.
    • Without the boundary, transcripts would 3–10× the parent's token bill.
  • Re-read the child's system prompt — "return ONE paragraph… do not narrate".
    • Sub-agents return artifacts, not history — the summary-not-state rule, enforced at the prompt layer.

The parent never sees the children's internal reasoning — each child starts with only its own system prompt and task brief. The summary-not-state rule comes from the companion study guide's Domain 1 material.

Tutorial 02 · Build-and-break

Forward the full child transcript up to the parent

  • Add spawn_subagent_LEAKY — it returns the child's full messages, not a summary string.
  • Flatten all three transcripts into the parent's history and synthesise from that.
  • Expected failure: the orchestrator's input_tokens roughly triples.
  • At N=10 the parent explodes its budget and starts losing the middle — Domain 5's failure mode.
  • Confirm: print resp.usage.input_tokens in both versions. Revert to the original spawn.

The leaky variant returns {"messages": [user brief, assistant reply]} and the orchestrator's messages becomes sum((f["messages"] for f in findings), []) — a flattened multi-conversation history masquerading as one conversation.

Tutorial 03 · Domain 2

Build an MCP Server Exposing Tools, Resources, and Prompts

Concept covered

  • MCP exposes three primitive types from one server — three control surfaces.
  • Tools are model-controlled, resources app-controlled, prompts user-controlled. Mixing up who controls what is Domain 2's most common exam trap.

You'll build

  • An MCP server exposing all three primitives
  • A stdio wiring into Claude Code
  • Each primitive fired by its correct initiator
  • A deliberate mis-categorisation, to watch it fail

Source: Domain 2, the MCP Triangle, plus the Tool / Resource / Prompt Template flashcards. Prereqs: Python 3.11+, pip install mcp (the official Model Context Protocol SDK), the Claude Code CLI (one-line native installer: curl -fsSL https://claude.ai/install.sh | bash), and basic JSON Schema familiarity.

Tutorial 03 · server.py

The tool — the model decides to call (1/2)

from mcp.server.fastmcp import FastMCP

mcp = FastMCP("triangle-demo")

# --- TOOL: model decides to call ------------------------------
@mcp.tool()
def lookup_order(order_id: str) -> dict:
    """Fetch current status of a customer order. Call only when
    the user references an order ID or asks 'where is my order'.
    Do not call for general product questions."""
    fake = {"A-100": {"status": "shipped"},
            "A-200": {"status": "processing"}}
    return fake.get(order_id, {"status": "not_found"})

Setup: mkdir mcp-triangle, fresh venv, pip install "mcp>=1.0". The decorator fires at import time — it introspects the function, pulls the docstring out as the description, and registers the tool in the server's capability table. The docstring is doing real routing work: when-to-call and when-not-to-call, in prose the model reads.

Tutorial 03 · server.py

The resource and the prompt (2/2)

# --- RESOURCE: app preloads, model reads ----------------------
@mcp.resource("docs://shipping-policy")
def shipping_policy() -> str:
    """Our shipping policy document. Read when explaining delivery times."""
    return "Orders ship in 2 business days. International adds 5–7 days."

# --- PROMPT: user types /refund -------------------------------
@mcp.prompt("refund-flow")
def refund_flow(order_id: str) -> str:
    """Walk the user through a refund for a specific order."""
    return (f"Start a refund flow for order {order_id}. Verify eligibility, "
            "confirm with the user, then call lookup_order to get status.")

mcp.run()   # defaults to stdio transport

mcp.run() starts the stdio event loop: the server blocks on stdin waiting for a JSON-RPC frame from Claude Code — JSON-RPC being the small request/response format MCP speaks over the pipe: a JSON object with a method name, params, and an id.

Tutorial 03 · Wiring

Wire it into Claude Code — .mcp.json

{
  "mcpServers": {
    "triangle": {
      "command": "python",
      "args": ["server.py"]
    }
  }
}

When Claude Code launches in this directory it reads .mcp.json, spawns python server.py as a subprocess, and handshakes over that subprocess's stdin/stdout.

After the handshake, Claude Code knows the server offers exactly one tool, one resource, one prompt — visible when you type /mcp. The file lives in the project root.

Tutorial 03 · Code trace

Three primitives, three initiator paths

  • TOOL — the model initiates. stop_reason=tool_use → Claude Code forwards a tools/call JSON-RPC frame over stdio → the result returns as a tool_result block. The description is the routing signal.
  • RESOURCE — the application initiates. Your @triangle:docs://… mention triggers resources/read; the content is inlined into the prompt. The model never emits a tool-use block.
  • PROMPT — the user initiates. /triangle:refund-flow is intercepted by the CLI, expanded via prompts/get, and submitted as if you had typed it.
  • Every MCP design choice reduces to: who should be in control of firing this?

Unlike earlier tutorials there is no single linear flow — that's the whole lesson. Claude Code implements app-control by letting you attach resources with @-mentions; other MCP hosts may attach relevant resources automatically. Either way the initiator is the application, not the model. Prompts don't change model behaviour — they change what the user has to type.

Tutorial 03 · Walkthrough

Fire each primitive from its own initiator

  • Boot check: python server.py < /dev/null — starts and waits on stdio; Ctrl-C to kill.
  • Launch claude, type /mcp — triangle appears with 1 tool, 1 resource, 1 prompt.
  • Ask "What's the status of order A-100?" — Claude calls lookup_order and answers "shipped".
  • Attach @triangle:docs://shipping-policy and ask about delivery — content attached, no tool call in the transcript.
  • Type /triangle:refund-flow — the template runs; the human decided when.

Typing @ lists every available resource. Stdio is the transport Claude Code uses for local MCP servers; the CLI auto-loads .mcp.json and introspects the server. In the refund flow the model will likely call lookup_order itself — a prompt composing with a tool.

Tutorial 03 · Build-and-break

Convert the shipping-policy resource into a tool

  • Rewrite it as get_shipping_policy() with a minimal description: "Returns shipping policy text."
  • Then ask a non-shipping question: "What's our return policy?"
  • Expected failures: a pointless call ("policy" matched loosely) — or no attach at all — or token bloat as the full document lands in every tool result.
  • Confirm: the spurious get_shipping_policy call in the transcript. A resource is attached up front; a tool demands the model find a reason.
  • Revert: restore the @mcp.resource decorator.

Mis-categorising a resource as a tool shifts the initiator from the app to the model — that's why it produces wrong-time calls and wasted tokens. The @-mention decides when a resource joins context; a tool's description has to argue for every single call.

Tutorial 03 · Verification checklist

Tick these only after you've seen them

  • I can state "who initiates?" for each of the three primitives without looking.
  • I've seen a tool mis-used because its description was too loose.
  • I can point to my .mcp.json and explain stdio vs Streamable HTTP transport.
  • My MCP server exposes one of each primitive and all register in /mcp.
  • I've watched Claude choose (or not) to call a tool based on its description.

Tool = model decides. Resource = app decides. Prompt = user decides. If you can answer "who initiates?" cold, the Domain 2 trap questions lose their teeth.

Tutorial 04 · Domain 2

Tool Description Discipline — Break It, Then Fix It

Concept covered

  • The description is the only signal the model has for when to call a tool — it routes by semantic match, not by function name.
  • Great descriptions state parameters, units, side effects, and anti-examples ("when NOT to call") — or an adjacent tool swallows the call.

You'll build

  • The same prompts against vague vs precise descriptions
  • A count of incorrect invocations per description
  • Proof that "when NOT to call" cuts false positives
  • The rename-to-refresh-attention anti-pattern, debunked

Three primitives, three initiators

  • HOOK — the harness initiates. A Write finishes → the PostToolUse matcher runs the shell command before control returns. The model cannot skip it — it didn't even know it ran. Deterministic → hook.

def extract_with_retry(doc: str) -> dict:
    messages = [{"role": "user",
                 "content": f"Extract the contact from:\n<doc>{doc}</doc>"}]
    for attempt in range(MAX_RETRIES):
        resp = client.messages.create(
            model=MODEL, max_tokens=512, tools=tools, messages=messages,
            tool_choice={"type": "tool", "name": "record_contact"},
        )
        tool_block = next(b for b in resp.content if b.type == "tool_use")
        errors = list(validator.iter_errors(tool_block.input))
        if not errors:
            return tool_block.input

python · tool_choice removes the model's option to reply with prose

Three design choices: tool use not prose (the input_schema is the target shape, so you never parse JSON out of prose); forced tool use via tool_choice (every response must contain a tool_use block); schema-level validation on top.

Tutorial 07 · Native structured outputs

The API now ships this natively — both GA on Sonnet 4.6

  • output_config={"format": {"type": "json_schema", "schema": …}} constrains the response itself — no tool needed; the SDK's client.messages.parse() validates into a Pydantic model.
  • strict: true on a tool makes the API guarantee that tool_use.input validates against input_schema (needs additionalProperties: false and a required list).
  • Reach for those first in production.
  • This tutorial still builds the loop: native output doesn't cover semantic constraints, and error-becomes-a-message generalises to every self-healing agent loop — what the exam probes.

Tutorial 07 · Code trace

The self-healing retry loop, attempt by attempt

  • Attempt 1 · happy path: the tool block validates clean — return on attempt 1, never loop.
  • Sad path: an ambiguous source ("about thirty-something") yields a type-valid guess or a string age — iter_errors returns a message and a JSON-pointer path.
  • The failing message becomes the next turn's input; is_error: True says "fix this", not "here's data".
  • Attempt 2: the model reads its own error and corrects the exact path — generic "try again" wastes the turn.
  • Attempt 3 exhausted: raise — never silently return bad data, never spin forever.

The retry message needs a concrete pointer to what failed — "Validation failed: 'age' is a required property. Path: []." — or the model has no signal and guesses a different wrong thing. Structured output = shape + semantics + retry.

One marker on the stable prefix

def ask(question, use_cache):
    system = [
        {"type": "text", "text": "You are a literary analyst."},
        {"type": "text", "text": f"<doc>{doc}</doc>",
         **({"cache_control": {"type": "ephemeral"}} if use_cache else {})},

    )

python · each call prints input_tokens, cache_read, cache_write, and wall time

The marker says: everything up to and including this block is a stable prefix — cache it under a key derived from the token contents. The script makes three calls: two with use_cache=True (miss, then hit) and a control without the marker. The trimmed print line reads the usage fields via getattr with a 0 default.

Tutorial 08 · Part A · Code trace

Three calls, three cache states

  • Call 1 · miss: prefix hashed, no entry — full processing plus a cache write. cache_creation_input_tokens ≈ 5000, at a ~25% write premium.
  • Call 2 · hit: same hash — ~5000 tokens served from cache at ~10% of the input rate. That's the whole reason to cache.
  • Call 3 · control: no marker — every token fresh, both counters zero. Your savings baseline.
  • The key is a hash of the exact prefix tokens — any edit, even punctuation, is a new key.
  • TTL: ephemeral entries live 5 minutes (1 hour premium). Insert time.sleep(310) and call 2 becomes a write again.

Why it matters: on a chatbot with a fixed 40K-token system prompt, nearly the whole prompt bills at a tenth the rate. Plan your wake-up intervals around the TTL — an expired entry is effectively a new key even though the prefix didn't change.

  • end: last thing read before the question — recency makes it almost as strong. Expect 3/3.
  • middle: buried in ~5000 tokens of narrative — expect a mix, often 2 of 3, worse on bigger documents.
  • Three trials because this is an unreliability pattern, not a hard failure — sometimes-fail is exactly why teams ship the bug.

This is the U-shaped attention curve from the Domain 5 source, now a number on your terminal rather than a diagram on a slide. If middle recall were always-fail the workaround would be obvious; because it's sometimes-fail, it survives code review.

Tutorial 08 · Build-and-break

Bury a critical instruction in the middle

  • Combine both failure modes: put "Before answering, always capitalise the first word" mid-document — and do NOT repeat it elsewhere.
  • Ask a question that requires following it.
  • Expected: intermittent compliance.
  • Fix per the Domain 5 "Do" list: repeat critical rules at start AND end — compliance jumps to ~100%.

System-prompt rules at position 0 aren't always enough on very long contexts — the anchor at the end is what closes the loop. That's the design rule this whole part exists to make visceral.

Tick these only after you've seen them

  • I've seen cache_read_input_tokens be non-zero on a repeat call.
  • I know the default ephemeral cache TTL (5 minutes).
  • I've measured middle-placement recall failing vs start/end.
  • I can explain why system-prompt rules at position 0 aren't always enough.
  • I've used cache_control on a stable prefix.

Troubleshooting

When things go wrong (1/2)

  • Could not resolve authentication method — export ANTHROPIC_API_KEY in the same shell, or load a .env via python-dotenv.
  • credit balance is too low — buy credit first, then mint a fresh key; a pre-credit key can stay stuck on this error.
  • ensurepip is not available — sudo apt install -y python3-venv.
  • python: command not found — Ubuntu ships python3; use it, or install python-is-python3.
  • externally-managed-environment — PEP 668: activate the tutorial's venv before pip install.
  • ModuleNotFoundError: anthropic — venv not activated; which python3 pip must point into .venv.

The errors most people hit running these tutorials, with the fix that actually works — especially on Ubuntu, where the defaults bite. The credit-balance gotcha is observed behaviour, not officially documented: the credit-check state attached to a key created before the account had credit doesn't always refresh. While in the console: keys must belong to a workspace that has balance, and auto-reload prevents mid-run balance drops in the fan-out tutorials (02, 07, 08). sudo also strips env vars — never run the scripts under sudo.

Troubleshooting

When things go wrong (2/2)

  • 404 model not found — pip install -U "anthropic>=0.40", confirm the ID against models.list(); Haiku 4.5 is the fallback.
  • 429 rate limit — new accounts sit on tight tiers; lower iteration caps or switch to Haiku while testing.
  • Tutorial 01 loop never exits — that's the intended break: Ctrl-C, restore MAX_ITERATIONS.
  • tool_result.content must be a string — json.dumps the result; never a raw dict.
  • claude: command not found after npm — npm's bin dir isn't on PATH; the native installer sidesteps this.
  • MCP server hangs on stdio — run python3 server.py directly to surface the traceback; check .mcp.json paths.

The SDK already retries transient 429s. The stdio hang is almost always a silent import error: Claude Code launches the server as a subprocess, so its stderr never surfaces in the CLI — the process dies quietly and the handshake never completes. If the direct run reports ready, check that the .mcp.json config path is absolute and the working directory is correct. Still stuck? Open a GitHub issue with the exact error and the command that produced it.
